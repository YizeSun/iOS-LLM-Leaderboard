# Power Benchmark 1.0 RC1 Submission Guide

## Status

This guide freezes the proposed repository submission package for
`suite-b-power@1.0.0-rc.1`. The package is executable and reviewable, but
public result intake and ranking remain closed until the maintainer explicitly
approves publication.

This path is separate from the historical `suite-b-community-submission-0.1`
Pilot wrapper. Pilot submissions are not promoted into Power 1.0 evidence.

## Package contents

One submission is exactly one App-exported result and one contributor-owned
manifest:

```text
submissions/suite-b/power-1.0.0-rc.1/draft/
└── <submissionID>/
    ├── submission.json
    └── result.json
```

- `result.json` is the exact, byte-for-byte App export. It must not be
  reformatted, corrected, trimmed, or regenerated by repository tooling.
- `submission.json` binds the result SHA-256 and public identities to the
  contributor's declarations.
- No screenshots, console logs, device names, account data, or other files are
  accepted in the package.
- The directory name must equal the UUID stored as `submissionID`.

The raw Power result already contains the token, timing, memory, thermal,
terminal-outcome, model, runtime, device-model, and OS evidence required for
independent recalculation. Additional private device identity is neither
needed nor allowed.

## Contributor declarations

Creating a package requires the contributor to affirm all of the following:

1. the run used a physical device;
2. the contributor is authorized to submit the evidence;
3. the contributor reviewed every public metadata field;
4. `result.json` is the unmodified App export;
5. the package contains no personal data;
6. benchmark evidence is contributed under CC BY 4.0; and
7. submission does not guarantee acceptance, verification, or ranking.

The contributor provides a public GitHub handle and a conflict-of-interest
category. Affiliation with a model, runtime, or hardware vendor is allowed but
must be disclosed; it does not make a result invalid by itself.

## Create a package

First inspect the App export. Then run:

```bash
python3 scripts/create_suite_b_power_submission.py \
  /path/to/app-export.json \
  --output-root submissions/suite-b/power-1.0.0-rc.1/draft \
  --contributor YOUR_GITHUB_HANDLE \
  --conflict-category none \
  --conflict-statement "No conflict of interest disclosed." \
  --accept-declarations
```

The tool:

- rejects a structurally invalid or protocol-incompatible result;
- copies the original result bytes unchanged;
- creates a new submission UUID;
- records exact result identities and SHA-256; and
- validates the completed package before returning its path.

`--accept-declarations` is an explicit legal and factual confirmation, not a
convenience default. Do not use it unless every declaration is true.

## Validate locally

Validate one package:

```bash
python3 scripts/validate_suite_b_power_submission.py \
  submissions/suite-b/power-1.0.0-rc.1/draft/<submissionID>
```

Validate the complete intake directory:

```bash
python3 scripts/validate_suite_b_power_submission.py \
  submissions/suite-b/power-1.0.0-rc.1/draft \
  --output /tmp/suite-b-power-intake-report.json
```

A valid Draft normally reports `validWithWarnings`: its bytes and protocol are
valid, while its evidence level remains `unreviewed` and ranking remains
unauthorized.

Metric eligibility is separate from package validity. A completed result may
validly retain failed, OOM, cancelled, not-run, response-ineligible, or
otherwise metric-ineligible attempts. Contributors must never rerun merely to
hide such evidence.

## Pull request and review

Open a pull request containing only the new package. CI reruns the frozen Power
result validator and package-integrity checks. Passing CI does not change the
evidence level.

A maintainer-created, hash-bound review record is required for every evidence
transition. Review records live under:

```text
submissions/suite-b/power-1.0.0-rc.1/reviews/
```

The App and contributor cannot assign `community-submitted`, `reproduced`,
`verified`, `maintainer-reference`, or default leaderboard eligibility. See
the [governance policy](power-benchmark-1.0-governance.md).

## Privacy boundary

Allowed public identity includes the model artifact and revision, runtime,
device machine identifier, OS version/build, physical-memory class, battery
state, thermal state, App source commit, result/session UUIDs, and the public
GitHub handle chosen by the contributor.

Do not submit Apple ID, email address, UDID, serial number, vendor identifier,
device-assigned name, host name, phone number, location, user documents,
personal prompts, local file paths, or unrelated logs. If such data is found,
stop review and follow the privacy-removal procedure in the governance policy.
